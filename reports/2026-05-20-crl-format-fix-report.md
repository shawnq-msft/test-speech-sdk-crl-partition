## Question

Why did CRL loading fail in the previous Speech SDK/OpenSSL reproduction, and can we fix/test it?

## Root cause

The mock CRL server was serving PEM-formatted CRLs at URLs ending in `.crl` with content type `application/pkix-crl`.

Evidence before the fix:

```text
$ file certs/partition1.crl certs/partition2.crl
certs/partition1.crl: ASCII text
certs/partition2.crl: ASCII text

$ openssl crl -inform DER -in certs/partition1.crl -noout -issuer
Could not read CRL from certs/partition1.crl
Unable to load CRL

$ openssl crl -in certs/partition1.crl -noout -issuer
issuer=CN = Test CRL Partition CA, O = Speech SDK CRL Test, C = US
```

Interpretation:

- `ASCII text` plus successful default `openssl crl -in ...` parsing means the generated file was PEM.
- DER parsing failed.
- The Azure Speech SDK native OpenSSL path downloads CRLs from CDP URLs and expects the HTTP `.crl` payload to be a DER-encoded CRL, which is the common format for `application/pkix-crl`.
- Because this harness served PEM bytes, the SDK downloaded a CRL but could not load it into OpenSSL, producing `Error loading CRL ...` and then `WS_OPEN_ERROR_UNDERLYING_IO_OPEN_FAILED`.

So the failure was a harness artifact: CRL encoding mismatch, not proof that a reachable valid CRL would fail.

## Fix applied

Updated `scripts/gen_crl.sh` to generate CRLs as PEM first, then convert to DER for the served `.crl` files:

```bash
openssl ca -gencrl \
    -config "${CNF}" \
    -out "${CERTS_DIR}/partition1.crl.pem" \
    -crldays 30
openssl crl \
    -in "${CERTS_DIR}/partition1.crl.pem" \
    -outform DER \
    -out "${CERTS_DIR}/partition1.crl"
```

The same pattern is used for `partition2.crl`.

## Result

After serving DER CRLs, CRL loading no longer fails.

Summary from the fixed run:

```text
initial_crl_enabled: status=connected elapsed=0.10160636901855469

after_rotation_crl_enabled: status=connected elapsed=0.1019446849822998

after_rotation_crl_disabled: status=connected elapsed=0.10100936889648438

✓ CRL checking enabled path connected; disable-CRL path also connected
```

Native SDK log evidence for CRL enabled path:

```text
tlsio_openssl.c:1882 CRL check enabled.
```

There are no `Error loading CRL` or `WS_OPEN_ERROR_UNDERLYING_IO_OPEN_FAILED` entries in the fixed CRL-enabled native log.

Native SDK log evidence for disabled path:

```text
name='OPENSSL_DISABLE_CRL_CHECK'; value='true'
tlsio_openssl.c:1878 CRL check off, as requested.
```

CRL server evidence:

```text
CRL server status after initial attempt:
  total_requests: 1
  partition requested: partition1

CRL server status after post-rotation CRL-enabled attempt:
  total_requests: 1
  blocked_partitions: [1]
```

The post-rotation connection still succeeded because the SDK used the cached CRL file from the same test CA. This harness currently models two URLs/partitions but both CRLs are generated by the same CA and have the same issuer/scope. With a valid cached CRL, OpenSSL accepts the certificate chain. To reproduce the Microsoft Learn `different CRL scope` class of failure more faithfully, the harness would need distinct CRL scopes/issuing distribution point extensions rather than just different URLs.

## Conclusion

- The immediate CRL loading failure was caused by serving PEM CRL bytes where the Speech SDK/OpenSSL CDP path expected DER CRL bytes.
- Generating served `.crl` files as DER fixes the loading failure.
- With the DER fix, Speech SDK 1.40.0 connects successfully with CRL checking enabled and also connects with `OPENSSL_DISABLE_CRL_CHECK=true`.
- The disable-CRL property is still verified: the native log explicitly shows `CRL check off, as requested`.
